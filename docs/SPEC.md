# 学习督促 Discord Bot · 产品需求文档

> 一份自包含的需求说明，读者无需任何前置背景。
> 最后更新：2026-06-27

**一句话概述**：一个常驻云端的 Discord Bot，读取使用者（下称「我」，一名学生）手写的学习计划，把每个 `study`/`work` 任务当作有生命周期的对象，在恰当时间主动在指定文字频道 @我 催我开始、盯我进度、确认我完成，并每天催我规划明天。

> 本文用到两个命名时刻——「日初」（每天的同步时刻）和「日末」（每天的收尾时刻）。它们在正文中只用名字指代，具体取值集中在第 7 节配置项。

---

## 1. 目标与用户

我有密集的备考和编程学习计划，容易拖延。我要一个像真人一样持续督促的工具：不是到点提醒一次就完，而是追踪每个任务从开始到完成的全过程。

- 我把计划写在 `plan.txt`，自然语言，格式随意，随时手动改。
- 我用 Discord，希望 Bot 在一个只有我俩的文字频道里主动 @我 催我。**用频道而非私信**：私信里 Discord 不允许我删掉 Bot 发的消息，而频道里我作为管理员可以随手删、保持整洁。
- 我不想改代码；运行期的调整通过编辑文件或发指令完成。
- Bot 部署在云端，24 小时运行。

## 2. 数据：两个文件

Bot 围绕两个文件运转：

| 文件 | 谁维护 | 作用 |
|------|--------|------|
| `plan.txt` | 我手写 | 原始计划，自然语言，行内格式随意（解析约定见下方注） |
| `progress.csv` | Bot 维护，我也可手动改 | 当天及以后任务的状态快照，可用表格软件直接打开 |
| `history.csv` | Bot 维护 | 昨天及更早任务的归档；每天「日初」从 `progress.csv` 移入，格式同 `progress.csv` |
| `context.txt` | 我手写，Bot 只读 | 给 Bot 的常驻自然语言说明：术语表 / 口径 / 语气偏好 / 关于我的背景，随时手改（见 4.5）|

> **`plan.txt` 的解析约定（只认行首）**：文件按「日期头」分节。**日期头** ＝ 以日期 `MM.DD` 开头的行（如 `07.01 周三`、`07.01 Game`）——只取其中的**日期**，后面的星期/科目备注是写给我自己看的，**不解析**。**任务行** ＝ 以时间 `HH:MM` 开头的行（如 `14:00 Game Chap1`），归到最近的日期头下，逐行全取（含吃饭/睡觉/起床等 `other`，它们界定时间边界，漏一个就毁掉排程）。**其余不以时间/日期开头的行 ＝ 备注**（如任务行下面缩进的「30min 刷题」「导图」）：由 LLM 归到它上面那个任务、填进该任务的 `notes` 列（见 §2 `notes` 行与 §9）；纯废话则丢弃。任务骨架（日期/时间/任务名/`subject`/`code`）由**确定性代码**解析、绝不漏行；**LLM 只在这份固定任务清单上标注 `type` 与 `notes`，无权增删改任务**。

> **`context.txt` 只影响「理解与措辞」，不进监工决策**：它拼进 LLM 的提示词，帮 Bot 读懂我的黑话、按我的口径说话（例：`plan.txt` 里写 `Game`，在 `context.txt` 里说明「Game = 博弈论课程，不是打游戏」，类型才不会判成 `other`、催促也不会说成「别打游戏」）。但监工**何时催 / 催谁 / 状态怎么推进**仍只看 `progress.csv` 快照 + 当前时间 + 第 7 节配置，**不读 `context.txt`**——与 3.2「监工无记忆」一致。所以把「10 点后别催我」「给我 30 分钟缓冲」这类**时间/机制**规则写进 `context.txt` 不会生效，那些要改配置或代码。Bot 从不回写此文件。

`progress.csv` 的列：

| 列 | 含义 |
|----|------|
| `code` | 唯一标识，格式 `date-time-subject`，如 `0313-1400-PGM`。`time` 是 4 位时钟 `HHMM`；**日初 4:00 之前的凌晨**（逻辑日的尾巴，见 §5）小时进位到 24–27，如 `01:00`→`2500`、`00:00`→`2400`——这样 code 的字典序 = 逻辑日顺序，行排序一律按 `code`（见 §3.2、§5），无需另算 |
| `date` | 如 `03.13` |
| `planned_time` | 头尾两端，如 `14:00-18:00`。头 = 本任务时间点；**尾 = 下一个任务的时间点**（注意：不是任务时长，而是下一个任务何时开始）|
| `task` | 任务描述 |
| `notes` | 该任务的备注（写在任务行下方的说明，如「30min 刷题；导图」），由 LLM 从 `plan.txt` 归集；**`plan.txt` 全权拥有**（见 3.1 例外），催促/查进度时作为「这任务具体要做什么」喂进文案 |
| `type` | `study` / `work` / `outing` / `other`，由 LLM 自动判断，我可手动改 |
| `status` | `todo` / `overdue` / `started` / `completed` |
| `actual_time` | 头 = 实际开始时间，尾 = 完成时间；`started` 时只有头，如 `14:08-` |
| `latest_progress` | 浓缩的进度描述（人读，可手动改）|
| `latest_progress_time` | 上面这条进度的时间戳，由 Bot 维护；监工去重只认这一列（与自由文本分开，手动改描述不会碰坏去重）|

任务行为按类型区分：

- **`study` / `work`**：进入第 4 节的完整生命周期，会被催。**例外**：若它正好是当天最后一个任务，则不催，只当时间边界（见 3.2、4.2、§7）。
- **`outing`**：预留类型，用于未来的「出门前提前提醒准备」功能（我容易迟到）；具体提醒逻辑暂未定义，当前仅占位、不触发任何动作。（未来实现「提前提醒」时需注意：提前量可能让它的提醒时刻落进前一个 `study`/`work` 任务的 `started` 区间，届时要协调这两条提醒——例如合并成一条、或让出门提醒优先。当前不触发动作，故暂无此问题。）
- **`other`**（吃饭、娱乐等）：不催，仅用来界定上一个任务的计划时间结尾。

## 3. 运行机制

### 3.1 凌晨快照

任务带状态（开始时间、进度、完成与否）。如果白天反复从 `plan.txt` 重新解析，会把这些状态冲掉，所以采用「每天同步一次」的方式：

- **每天「日初」**，按顺序做两件事：
  1. **归档**：把 `progress.csv` 里**昨天及更早**逻辑日（见第 5 节）的行移进 `history.csv`，`progress.csv` 只留今天及以后的任务，避免无限累积。
  2. **同步**：从 `plan.txt` 解析**今天及以后**的任务，与 `progress.csv` 现有行做一次集合对比（见下方「同步＝集合对比」）。
- **白天**：只读 `progress.csv`，不再碰 `plan.txt`。
- 我改了 `plan.txt` 想立即同步，可发 `!sync`：跑同一套集合对比（但**不归档**，归档只在日初做）。

**同步 ＝ 集合对比（日初与 `!sync` 共用）**——以 **`code`** 为身份，比对「`plan.txt` 解析出的今天及以后任务」与「`progress.csv` 里今天及以后的行」：

- 文档有、表格无 → **新增**该行（此时——也仅此一次——由 LLM 判断 `type`）。
- 表格有、文档无 → **删除**该行。
- 两边 `code` 都有 → **整行不动**（`status`、`actual_time`、`latest_progress`、`type` 等全部保留，不重判、不刷新）。**唯一例外：`notes` 列**——见下。
- **今天以前的行一律不参与对比、不被增删。**

如此匹配上的任务永不被改写，「保留状态」无需逐列讨论；`type` 只在新增时判一次，之后任我手动改、再同步也不覆盖。

> **`notes` 是「整行不动」的唯一例外**：`plan.txt` 是备注的唯一主人，所以同步时即便 `code` 匹配、其它列全不动，**`notes` 也会从 `plan.txt` 重新刷一遍**（我在 `plan.txt` 加/改/删某任务的备注，下次同步就跟着变；删了就清空）。反过来：**我直接在 `progress.csv` 手改的 `notes` 会被下次同步覆盖**——想改备注就改 `plan.txt`。这与「不盖手动编辑」的初衷不冲突：`notes` 本就声明由 `plan.txt` 拥有，其余运行期列仍照旧受保护。`notes` 不进监工决策（监工只看 `status` + 时间），刷不刷都不影响催促时机。

> **`code` 会变要留意**：`code` = `date-time-subject`，所以在 `plan.txt` 里改了某任务的时间或科目，`code` 随之变化，集合对比会视为「删旧 `code` + 加新 `code`」，那一行原有的运行期状态（开始时间、进度）会随旧行一起没掉。改没开始的任务无妨；改 `started` 的任务前先想清楚，或事后在 `progress.csv` 手动补回。
>
> **`code` 撞车 = 文档 typo**：正常情况下 `code` 天然唯一；若解析 `plan.txt` 时两条任务算出同一 `code`，一定是我把文档写重了——Bot **反问让我改 `plan.txt`**，不自行猜测处理。

### 3.2 监工轮询（核心机制）

Bot 不为每个任务设固定闹钟。取而代之，一个「监工」**每分钟运行一次**，读取 `progress.csv` 的当前内容。它**没有记忆**，不比较「和上一分钟有什么不同」——只看每个任务此刻的 `status` 与当前时间，据此决定这一刻该做什么。

对每个 `study`/`work` 类型、属于**当前逻辑日**（见第 5 节）、且**不是当天最后一个任务**的任务，监工的决策如下（当天最后一个任务一律不催，只当时间边界，见 4.2 与第 7 节）：

| 看到的状态 | 时间情况 | 动作 |
|------------|----------|------|
| `completed` | — | 跳过 |
| `todo` | 未到计划开始时间 | 等待 |
| `todo` | 已到计划开始、未过计划结尾 | 催开始（首次）→ 把状态写成 `overdue` |
| `todo` | 已过计划结尾（时间窗整个错过） | **不催**（催「现在开始」已无意义）；静默转 `overdue`，留到「日末」总结——与 `overdue` 过结尾同理 |
| `overdue` | 未过计划结尾 | 到退避曲线的催促时刻就催一次（间隔随拖延递增，见 4.1 与配置项），直到我确认开始 |
| `overdue` | 已过计划结尾 | 停止催促；状态保持 `overdue`（表示我始终没开始），留到「日末」总结 |
| `started` | 未到中点 | 等待 |
| `started` | 已过中点**但未过结尾**，且 `latest_progress_time` 早于中点 | 发中点进度确认，并随即写占位（见下方「健壮性」）|
| `started` | 已过结尾，且 `latest_progress_time` 早于结尾 | 发结尾进度确认 + 任务交接，并随即写占位；状态保持 `started`，留到「日末」总结 |

（中点 = 计划时间头、尾的中点。**结尾优先于中点**：宕机重启后若中点、结尾都已过，只走结尾交接，不再补问那个早已过去的中点——上表两条 `started` 行已用「未过结尾」互斥，不会同时触发。）

**监工为何无需记忆**：它判断「这分钟该不该发中点提醒」，不靠记住发没发过，而是比对 `latest_progress_time`——若它已晚于中点，说明问过了，不再重复；结尾同理。因此它永远只是「读当前快照、下当前结论」，重启也不会错乱。

**手动编辑为何即时生效**：监工只认文件此刻的内容，所以我手动改 `progress.csv` 就等于改了它下一分钟看到的状态。把某行状态改成 `completed`，它下一分钟便跳过；删掉某行，它下一分钟便看不到。**文件是唯一事实来源，监工绝不用内存里的旧数据覆盖文件。** 为免覆盖我正在外部编辑器里保存的改动，监工写回时**先重读文件、只改它这一轮真正要动的那几格（行），不整文件覆盖**——这样我手动编辑的其它行不会被它的旧快照冲掉。（换 `STORAGE_BACKEND=notion` 时，「文件」泛化为「外部权威存储」——事实来源是 Notion 库，监工每 tick 重读、按 `code` 只 PATCH 这一行，我在 Notion 里的手改一样即时生效，见 §9。）

**健壮性：不依赖我及时回应**。Bot 的任何环节都不会因为我不回、晚回而出错或卡住：

- 发出中点 / 结尾进度确认的**同时**，就写一条占位：`latest_progress` 填「等待回应」、`latest_progress_time` 填当前时刻。这样去重立即生效，**不会因为我没回复而每分钟重复追问**；等我真回复了，占位被实际进度替换、时间戳一并更新。
- 催促开始时，若到该任务的计划时间结尾我仍未开始，就**停止催促**（状态保持 `overdue`，作为没开始的任务留到「日末」总结），不会无限催下去。
- 总原则：我不回应时，系统只是**保持当前状态**、必要时用「等待回应」占位标注，绝不重复轰炸、也绝不卡死。

## 4. 任务生命周期

每个 `study`/`work` 任务是一个独立的状态机；多个任务可同时处于不同状态。

```
todo ─(到点)→ overdue ─(确认开始)→ started ─(确认完成)→ completed
                       ↑每分钟催               ↑中点/结尾查进度
```

状态的推进由两类事件触发：
- **监工**：到点把 `todo` 推到 `overdue`。
- **我的回复 / LLM 判断**：把 `overdue` 推到 `started`，把 `started` 推到 `completed`。

### 4.1 催促开始（状态 = `overdue`）

- 到点后开始催，**每条消息都用模糊指代点明是哪个任务**（不必照抄 `code`，能让我认出即可，如「PGM 那个」），因为可能多个任务并行。
- **间隔随拖延递增（退避）**：刚到点时催得密（每分钟一次），我越拖、间隔拉得越长，免得一小时堆几十条消息。监工靠「当前距计划开始多久」来决定这一分钟要不要催，并不需要记住上次催的时间；默认曲线见第 7 节。
- **强度随拖延递增**：刚到点温和，越拖越硬。语气的完整规则（性格 × 强度）见 4.5。
- **每条催促都给可执行的启动指示（ADHD 友好、与性格无关）**：我有严重的启动/切换困难，所以不管抽到什么性格，催开始的消息都要给一个**具体、可做的第一步**并**要求我汇报**（开始了吗 / 在做什么），而不只是喊「快开始」。**首次**催再附一份**实用**的「如何切进并开始」拆解，拆成 **3–5 个小步**（给明确步数，别用「几步」「简洁」这种模糊词——LLM 拿不准；上限 5 步，别繁杂到我也做不到）；**之后**的催保持轻量，只帮我迈出第一步。首次 vs 之后由状态区分（这轮才从 `todo` 翻成 `overdue` 的是首次），监工无需记忆。
- **真实性判定门槛高**：只有回复**明确显示我已进入专注**才算数；「好」「在了」「马上」这类敷衍/应付**一律判否**（拿不准也判否）。判否时不额外追问，继续按上面的方式催（催促本身已在要我汇报）。
- **退出**：我发关键词 `!started <模糊 code>`，或 LLM **高置信**判定我确实开始了 → 状态改 `started`，记录实际开始时间，并把 `latest_progress_time` 初始化为该实际开始时间（开始必早于中点，故中点去重的「早于中点」天然成立，无需处理空值）。到该任务**计划结尾**仍没开始则按 3.2 停催（状态保持 `overdue`，留到日末总结）——这条安全阀不变。

### 4.2 盯进度（状态 = `started`）

- 在**计划时间中点**发一次进度确认，把我的回复浓缩进 `latest_progress`、并把 `latest_progress_time` 更新为当前时刻。**只记实质进度**：仅当回复带具体内容（做了什么 / 到哪了 / 卡在哪）才浓缩记录；单纯的应答、承诺、表态、情绪、要我别阴阳等一律按纯聊天处理、不写文件——免得一句空泛的「我好好写」盖掉已有的「写了一半」。
- 在**计划时间结尾**再确认一次，并做**任务交接**：
  - 我发 `!completed <模糊 code>`（或 LLM 判断已完成）→ 状态改 `completed`。
  - 我仍在描述进度 → 浓缩更新 `latest_progress`。
  - **无论是否完成，此后都不再单独催这个任务。** 若下一个任务是 `study`/`work` 类型、**且不是当天最后一个任务**，就在**同一条消息**里回顾本任务进度、宣布切换、开始催下一个（避免发两条割裂的消息）。这条交接消息**即充当下一个任务的首次催促**——发出后把下一个任务直接置为 `overdue`，监工便不会再对它单独重发首次催，因此不会重复。若下一个任务正是当天最后一个（一般是睡觉），则只回顾本任务、不催它。
  - **宕机重启的兜底（防双重催）**：重启后同一轮里可能既要给当前任务做结尾交接、下一任务又已过开始时间。监工**按计划时间从早到晚处理**；做交接前先看下一任务是否已被置 `overdue`，已是就**不在交接里重复它的首催**——避免「监工首催 + 交接首催」把它催两遍。
- 到结尾仍未完成的任务**不滚动追问**，统一留到当天「日末」的总结里列出。

### 4.3 真实性判断的边界

催促开始与盯进度时，Bot 都靠 LLM 读我的回复来判断我是真在做事还是敷衍。Bot**没有任何独立监测手段**（如摄像头），只能依据对话内容，所以判断不保证准确。我可以随时用关键词或直接编辑文件来纠正状态。

### 4.4 多个任务同时活跃时

我尽量一次只推进一个任务，但偶尔会有重叠。设计本身允许并行——监工对每个任务独立判断，每条催促也都标明是哪个任务——但有几点必须注意：

- **回复归属（重要）**：当多个任务同时活跃、而我的回复没点明是哪个时（如只说「做完了」），若存在多个可能的匹配，Bot**反问澄清是哪个任务，绝不猜着改状态**。我想省事就直接用 `!started / !completed <模糊 code>` 指明。
- **避免轰炸**：同一分钟若有多个任务都该发消息，合并成一条发出，不连发数条。这条合并消息的**性格取当前最拖延的那个任务**的性格（拖得最久的优先定调，见 4.5）。
- **计划时间假设不重叠**：「计划时间」的尾取自下一个任务的时间点，这隐含任务按时间首尾相接。若我在 `plan.txt` 里写了时间重叠的任务，中点/结尾/交接的计算会失准——尽量避免，或在 `progress.csv` 里手动修正。
- **`outing` 与 `study`/`work` 撞车（未来）**：`outing` 要提前催「准备出门」，可能正好撞上某个 `started` 的 `study`/`work` 任务。该功能实现时需定义两者的优先级或合并方式；当前 `outing` 不触发，暂无此冲突。

### 4.5 消息语气

每条发给我的消息，语气由两个独立维度合成：

- **性格**：温柔鼓励 / 严厉施压 / 阴阳怪气 三选一，决定说话风味。默认**每个任务固定一个性格**（这个任务的所有消息都用它，不会忽冷忽热）；可在配置项一键切换成「每条消息重抽」或「每天固定一个」。**多任务合并成一条时**（见 4.4），整条用**当前最拖延的那个任务**的性格，不混搭。
- **强度**：只在催促开始阶段随拖延时长递增（越拖越硬）；其余消息（进度确认、交接、日末总结、闲聊）用常态强度。

所以一条催促开始的消息 = 该任务的性格 + 随拖延递增的强度，例如「越来越辛辣的阴阳怪气」或「越来越急的温柔」。

**文案会参考最近若干条频道消息作上下文**：生成任何消息时都带上最近的对话，好让 Bot 领会我的即时纠正/口径——比如任务写「写单元测试」它起初理解成编程，我纠正「这是考试、不是编程」后，之后的催促就该跟着改口，而不是每分钟从头失忆、重复错误指示。**注意边界**：这只影响**怎么说**（文案措辞）；监工的**决策**（何时催、催谁、状态怎么推进）仍然只看 `progress.csv` 快照 + 当前时间，不靠对话记忆——与 3.2 的「监工无记忆」不冲突（对话记录在 Discord 里，重启也不丢，故无需 Bot 自己存）。

**外加一份常驻说明 `context.txt`**（见第 2 节）：最近对话是**即时**纠正、会滑出窗口，`context.txt` 是我手写的**长期**口径，不会过期。两者互补，一并拼进 LLM 提示词，影响 Bot 对我话语的**理解**（术语、黑话、「我说'收工'就是完成」这类口径）和**措辞**（语气偏好、关于我的背景）——覆盖所有解读任务含义的 LLM 环节（解析计划、判断真实性、匹配模糊指代、写文案）。同样只作用于「理解 + 怎么说」，绝不改变监工决策；边界详见第 2 节 `context.txt` 说明。

## 5. 每日节奏

```
「日初」同步  →  白天各任务跑生命周期  →  「日末」总结 + 催计划  →（次日循环）
```

**逻辑日（作息日）**：一天的边界不在午夜，而在**「日初」时刻**（默认 4:00）。`[日初, 次日日初)` 这段就是一个「逻辑日」。任务的逻辑日 = 它在 `plan.txt` 里写的日期；凡钟点时间**早于日初**的（如 `01:00`），都算作该逻辑日的**后半夜**，构造绝对时间时落在「日期 +1 个日历日」。于是 `23:00-01:00` 这种跨午夜任务：头 = 当天 23:00、尾 = 次日 01:00、中点 = 次日 00:00，连续可比。监工说的「属于今天」= 任务逻辑日 == 当前逻辑日（现在早于日初则属昨天，否则属今天）。因为「日末」最晚不超过日初（见第 7 节），一个逻辑日正好装下含后半夜的全部任务。

**行排序一律按 `code`**：`code` 的时间字段已把后半夜的钟点进位到 24–27（见第 2 节，`01:00`→`2500`），所以 `code` 的字典序天然 = 逻辑日顺序——后半夜任务排到当天末尾。存储写回、`!progress` 表格、监工遍历都直接按 `code` 排，不再单算逻辑日分钟。本地文件与 Notion 表用同一个排序键，所以 Notion 视图只要设成按 `code` 升序，就能复现完全一致的顺序。

**「日末」**，Bot 做两件事：
1. **当日总结**：列出今天哪些任务完成、哪些没完成。
2. **催写计划**：直接催我写好/补全明天的计划（不检查 `plan.txt` 里现有内容，直接催）。我写好后，次日「日初」自动同步进 `progress.csv`。

## 6. 我的指令

| 输入 | 作用 |
|------|------|
| `!started <模糊 code>` | 把任务标记为 `started`，停止催开始 |
| `!completed <模糊 code>` | 把任务标记为 `completed` |
| `!sync` | 从 `plan.txt` 重新同步到 `progress.csv`：以 `code` 做集合对比，新增缺的、删除多的、匹配上的整行不动（见 3.1）；不归档。**同步成功后回一条成功消息，紧接着渲染当前表格（等同 `!progress`），好让我立刻看到结果**；解析报错（如 `code` 撞车）时只报错、不渲染 |
| `!progress` | 把 `progress.csv` 渲染成表格发给我：默认只出第一列（状态 emoji + 时间 + 任务）；`!progress detailed` 才多出第二列进度（`# 进度`）|
| `!whattodo [模糊 code]` | 把任务拆成 3-5 个「马上能做」的小步(复用催开始的首次拆解,见 4.1),帮我迈出第一步;不带参 = 当前正处于时间窗内的任务,带参 = 我指代的那个任务(带 `notes` 一起考虑)。**只读,不改任何文件** |
| `!modify` | 让 Bot 修改 `progress.csv`（删任务、改时间等；只动这一个文件）|
| `!clear [N｜时长｜all]` | 删 Bot 自己发过的消息（不带参=最近 1 条；`N`=最近 N 条；`30m`/`2h`/`1d`=最近该时段内；`all`=全部）|
| `!reloadcontext` | 重新载入 `context.txt`（术语/口径说明）。仅 `notion` 模式有意义：context 平时缓存，改完 Notion 页面发此指令即时生效（否则次日日初才刷）；`file` 模式本就即时，此指令为空操作 |
| 其它任何消息 | 纯聊天，结合计划与进度回复，**不改任何文件** |

**模糊 code**：用 `!started` / `!completed` 时不必逐字照抄 `code`，写个能认出是哪个任务的说法即可（如 `!completed PGM`、`!completed 下午那个`），由 LLM 匹配到具体任务。多任务并行、指代不清时，Bot 会反问而不是猜（见 4.4）。

**`!clear` 的边界**：`!clear` 按作者过滤，只删 **Bot 自己发的**消息，删不到我发的——删我发的需要「管理消息」权限，Bot 不授予；但我自己在频道里可以随手删任何消息（这正是用频道而非私信的好处）。Bot 逐条删（有速率限制，条数多时会删一会儿）；删除不可逆，只用于清理，不动 `progress.csv` 等任何数据文件。

**`progress.csv` 何时会被改动**——仅三种情况：① 我手动编辑文件；② 我用了上表中会改文件的指令（`!started`、`!completed`、`!sync`、`!modify`）；③ LLM 从对话中判断我确实开始/完成了某任务，自动改其状态——单任务场景**直接改、不再回头跟我确认**（接受偶发误判，反正我能随时手动纠正）；仅多任务歧义、指代不清时才先反问（见 4.4）。除此之外的对话一律是纯聊天，Bot 不擅自改文件。

## 7. 配置项（含默认值）

| 项 | 默认 |
|----|------|
| 名字（Bot 怎么自称）| `Bot`（我可以给它起名）|
| 时区 | `Europe/London`（用 IANA 时区名，**自动处理英国夏令时/冬令时**：夏季 BST=UTC+1、冬季 GMT=UTC+0 由时区库按日期自动切换，不写死偏移）。所有墙钟时间、日初/日末、逻辑日边界、退避计时都按它解释。实现须用 tz-aware 的 datetime（如 `zoneinfo`），切换日的「缺一小时/多一小时」自然由它处理 |
| 日初（每天首次运行：从 `plan.txt` 同步到 `progress.csv` 的时刻）| 4:00（环境变量 `DAY_START`）|
| 日末（每天收尾：当日总结 + 催写明天计划的时刻）| 23:00（可往后调，但**最晚不超过日初 4:00**；它是逻辑日的收尾，见第 5 节；环境变量 `DAY_END`）|
| 催促退避曲线（间隔随拖延递增）| 设 `m` = 距计划开始的整分钟数：`m<5` 每分钟催（0,1,2,3,4）；`5≤m<15` 锚 5、每 3 分（5,8,11,14）；`15≤m<30` 锚 15、每 5 分（15,20,25）；`m≥30` 锚 30、每 10 分（30,40,50…）。**每阶段首分钟必催、阶段切换无空档**；纯函数判定，监工无需记忆 |
| 监工轮询间隔 | 1 分钟（环境变量 `POLL_SECONDS`，单位秒）|
| 当天最后一个任务 | **一律不催**，只用它的开始时间界定倒数第二个任务的结尾（一般是睡觉）。想让某个 `study`/`work` 任务被催，就别把它放在最后——在它后面再写一个收尾任务 |
| 性格粒度 | 默认「每个任务一个」；可切换「每条消息一个」或「每天一个」（环境变量 `PERSONALITY_GRANULARITY`：`per_task` / `per_message` / `per_day`）|
| 文案参考的对话条数 | 12（生成消息时带上最近 N 条频道消息作上下文，好领会我的即时纠正，见 4.5；环境变量 `DIALOGUE_HISTORY`）|

## 9. 实现概要（技术选型，可替换）

**语言与布局**：Python 3.12+（3.13 可）。`src/` 布局，配置统一进 `pyproject.toml`，代码写类型注解。

**开发工具链**（2026 主流默认；若项目已在用别的就跟项目走，不擅自迁移）：

- 环境与依赖：**uv**（`uv add` 装依赖、`uv run` 跑命令、`uv sync` 同步环境；不手搓 venv、不直接 `pip install`）。
- Lint + 格式化：**ruff**（`ruff check --fix`、`ruff format`，一并替掉 black / flake8 / isort）。
- 测试：**pytest**。
- 类型检查：**mypy `--strict`**。（`ty` 很有前景但仍 beta、第三方 stub 不全，可本地试跑，CI 仍以 mypy 为准。`discord.py` 等库的 stub 不全时，局部 `# type: ignore` 即可，不必死磕。）
- 原则：能用「标准库 + 上面这套」解决就不加新依赖；其它库只在确实需要时才引入。

**运行时依赖（刻意压到最少）**：

- `discord.py` —— 在指定文字频道收发消息，并用它自带的 `ext.tasks`（`@tasks.loop(minutes=1)`）来实现「监工」每分钟轮询，以及「日初」「日末」的定时触发；这样**省掉 APScheduler**。
- 一个 LLM 客户端（`openai` SDK 调 DeepSeek / Claude 的兼容接口，或直接 `httpx`）。
- CSV 用**标准库 `csv`**；配置用**环境变量 + 一个 dataclass**（不上 pydantic）。

**各部分落地**：

- 通信：Discord Bot，发到配置指定的文字频道（`DISCORD_CHANNEL_ID`）；消息 @我 以触发通知；只处理来自我、且来自该频道的消息。
- 调度：`ext.tasks` 每分钟跑监工；日初 / 日末靠时间判断触发。
- 解析：`plan.txt` 的**任务骨架**（日期/时间/任务名/`subject`/`code`）由**确定性纯函数**解析（`plan_parser.parse_plan_text`），不经 LLM，绝不漏行、`code` 稳定；LLM 只在这份固定清单上**标注** `type` 与 `notes`（读整段 plan、闭集不可增删，见 §2 解析约定与 3.1）。
- 智能：其余判断调 LLM——标注 `type`/`notes`、判断我回复的真实性、把进度浓缩成一句、生成语气文案（性格 × 强度，见 4.5）、把「模糊 code」匹配到具体任务（歧义则反问，见 4.4）。
- 存储：默认本地文件——`plan.txt` + `progress.csv` + `history.csv`（归档）+ `context.txt`（我手写的只读说明，见第 2 节）。可选 `STORAGE_BACKEND=notion` 把这四者放到 Notion（progress/history 合成一个数据库、用 `archived` 复选框 + 视图分「今天 / 历史」；plan、context 各一个页面），由 `storage/` 后端抽象（`build_backend` 选择）封装。监工语义与所有不变式不变——「外部权威存储」仍是唯一事实来源；后端只换介质，不换规则。

**测试重点**：监工的决策（3.2 节那张「状态 + 时间 → 动作」表）是纯函数式、不碰 IO，最适合单元测试。本文一路记录的边界情形——中点 / 结尾不重复询问、催促到结尾即停（不无限催）、交接不重复催下一个、不依赖我及时回应——每条都应写成测试钉死，日后改动若破坏其一会立刻报警。CSV 读写一并测。Discord / LLM 这类 IO 难测，mock 或暂不覆盖。

**部署约束（重要坑）**：`progress.csv` 必须放在**重启后不会丢的存储**上。许多免费托管平台的文件系统是临时的（ephemeral），一重启就清空——若状态文件存在那里，Bot 半夜重启一次，所有任务的开始 / 进度 / 完成状态就全没了。务必用平台提供的**持久卷**（如 Railway Volume、Render Disk），或用文件不会丢的 VPS。（用 `STORAGE_BACKEND=notion` 则天然免除此坑：状态存在 Notion，重启不丢——只是 Bot 运行期需能联网访问 Notion。）
