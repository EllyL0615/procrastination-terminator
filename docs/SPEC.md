# 学习督促 Discord Bot · 产品需求文档

> 一份自包含的需求说明，读者无需任何前置背景。
> 最后更新：2026-06-27

**一句话概述**：一个常驻云端的 Discord Bot，读取使用者（下称「我」，一名学生）手写的学习计划，把每个学习/工作任务当作有生命周期的对象，在恰当时间主动私信催我开始、盯我进度、确认我完成，并每天催我规划明天。

> 本文用到两个命名时刻——「日初」（每天的同步时刻）和「日末」（每天的收尾时刻）。它们在正文中只用名字指代，具体取值集中在第 7 节配置项。

---

## 1. 目标与用户

我有密集的备考和编程学习计划，容易拖延。我要一个像真人一样持续督促的工具：不是到点提醒一次就完，而是追踪每个任务从开始到完成的全过程。

- 我把计划写在 `plan.txt`，自然语言，格式随意，随时手动改。
- 我用 Discord，希望 Bot 主动发私信催我。
- 我不想改代码；运行期的调整通过编辑文件或发指令完成。
- Bot 部署在云端，24 小时运行。

---

## 2. 数据：两个文件

Bot 围绕两个文件运转：

| 文件 | 谁维护 | 作用 |
|------|--------|------|
| `plan.txt` | 我手写 | 原始计划，自然语言，格式随意 |
| `progress.csv` | Bot 维护，我也可手动改 | 每个任务的状态快照，可用表格软件直接打开 |

`progress.csv` 的列：

| 列 | 含义 |
|----|------|
| 代号 | 唯一标识，格式 `日期-时间-科目`，如 `0313-1400-PGM` |
| 日期 | 如 `03.13` |
| 计划时间 | 头尾两端，如 `14:00-18:00`。头 = 本任务时间点；**尾 = 下一个任务的时间点**（注意：不是任务时长，而是下一个任务何时开始）|
| 任务 | 任务描述 |
| 类型 | `学习` / `工作` / `外出` / `其他`，由 LLM 自动判断，我可手动改 |
| 状态 | `未开始` / `催促中` / `进行中` / `已完成` |
| 实际时间 | 头 = 实际开始时间，尾 = 完成时间；进行中时只有头，如 `14:08-` |
| 最新进度 | 浓缩的进度描述 + 时间戳 |

任务行为按类型区分：

- **学习 / 工作**：进入第 4 节的完整生命周期，会被催。
- **外出**：预留类型，用于未来的「出门前提前提醒准备」功能（我容易迟到）；具体提醒逻辑暂未定义，当前仅占位、不触发任何动作。（未来实现「提前提醒」时需注意：提前量可能让它的提醒时刻落进前一个学习/工作任务的进行中区间，届时要协调这两条提醒——例如合并成一条、或让出门提醒优先。当前不触发动作，故暂无此问题。）
- **其他**（吃饭、娱乐等）：不催，仅用来界定上一个任务的计划时间结尾。

## 3. 运行机制

### 3.1 凌晨快照

任务带状态（开始时间、进度、完成与否）。如果白天反复从 `plan.txt` 重新解析，会把这些状态冲掉，所以采用「每天同步一次」的方式：

- **每天「日初」**：读 `plan.txt` → 解析当天任务 → 写入 `progress.csv`（新任务建档，已存在的任务保留其状态）。
- **白天**：只读 `progress.csv`，不再碰 `plan.txt`。
- 我改了 `plan.txt` 想立即同步，可发 `!重载`（同样保留已有状态，只新增/更新）。

### 3.2 监工轮询（核心机制）

Bot 不为每个任务设固定闹钟。取而代之，一个「监工」**每分钟运行一次**，读取 `progress.csv` 的当前内容。它**没有记忆**，不比较「和上一分钟有什么不同」——只看每个任务此刻的**状态**与当前时间，据此决定这一刻该做什么。

对每个「学习/工作」类型、且属于今天的任务，监工的决策如下：

| 看到的状态 | 时间情况 | 动作 |
|------------|----------|------|
| 已完成 | — | 跳过 |
| 未开始 | 未到计划开始时间 | 等待 |
| 未开始 | 已到计划开始时间 | 催开始 → 把状态写成「催促中」|
| 催促中 | 未过计划结尾 | 到退避曲线的催促时刻就催一次（间隔随拖延递增，见 4.1 与配置项），直到我确认开始 |
| 催促中 | 已过计划结尾 | 停止催促；状态保持「催促中」（表示我始终没开始），留到「日末」总结 |
| 进行中 | 未到中点 | 等待 |
| 进行中 | 已过中点，且「最新进度」时间戳早于中点 | 发中点进度确认，并随即写占位（见下方「健壮性」）|
| 进行中 | 已过结尾，且「最新进度」时间戳早于结尾 | 发结尾进度确认 + 任务交接，并随即写占位；状态保持「进行中」，留到「日末」总结 |

（中点 = 计划时间头、尾的中点。）

**监工为何无需记忆**：它判断「这分钟该不该发中点提醒」，不靠记住发没发过，而是比对「最新进度」的时间戳——若最新进度已晚于中点，说明问过了，不再重复；结尾同理。因此它永远只是「读当前快照、下当前结论」，重启也不会错乱。

**手动编辑为何即时生效**：监工只认文件此刻的内容，所以我手动改 `progress.csv` 就等于改了它下一分钟看到的状态。把某行状态改成「已完成」，它下一分钟便跳过；删掉某行，它下一分钟便看不到。**文件是唯一事实来源，监工绝不用内存里的旧数据覆盖文件。**

**健壮性：不依赖我及时回应**。Bot 的任何环节都不会因为我不回、晚回而出错或卡住：

- 发出中点 / 结尾进度确认的**同时**，就往「最新进度」写一条带当前时间戳的占位（如「等待回应」）。这样去重立即生效，**不会因为我没回复而每分钟重复追问**；等我真回复了，占位被实际进度替换。
- 催促开始时，若到该任务的计划时间结尾我仍未开始，就**停止催促**（状态保持「催促中」，作为未开始的任务留到「日末」总结），不会无限催下去。
- 总原则：我不回应时，系统只是**保持当前状态**、必要时用「等待回应」占位标注，绝不重复轰炸、也绝不卡死。

## 4. 任务生命周期

每个学习/工作任务是一个独立的状态机；多个任务可同时处于不同状态。

```
未开始 ─(到点)→ 催促中 ─(确认开始)→ 进行中 ─(确认完成)→ 已完成
                  ↑每分钟催            ↑中点/结尾查进度
```

状态的推进由两类事件触发：
- **监工**：到点把「未开始」推到「催促中」。
- **我的回复 / LLM 判断**：把「催促中」推到「进行中」，把「进行中」推到「已完成」。

### 4.1 催促开始（状态 = 催促中）

- 到点后开始催，**每条消息都用模糊指代点明是哪个任务**（不必照抄代号，能让我认出即可，如「PGM 那个」），因为可能多个任务并行。
- **间隔随拖延递增（退避）**：刚到点时催得密（每分钟一次），我越拖、间隔拉得越长，免得一小时堆几十条消息。监工靠「当前距计划开始多久」来决定这一分钟要不要催，并不需要记住上次催的时间；默认曲线见第 7 节。
- **强度随拖延递增**：刚到点温和，越拖越硬。语气的完整规则（性格 × 强度）见 4.5。
- LLM 判断我回复的真实性：若像敷衍（「好」「在了」「马上」），追问我是否真的开始了。
- **退出**：我发关键词 `!已开始 <模糊代号>`，或 LLM 判断我确实开始了 → 状态改「进行中」，记录实际开始时间。

### 4.2 盯进度（状态 = 进行中）

- 在**计划时间中点**发一次进度确认，把我的回复浓缩进「最新进度」。
- 在**计划时间结尾**再确认一次，并做**任务交接**：
  - 我发 `!已完成 <模糊代号>`（或 LLM 判断已完成）→ 状态改「已完成」。
  - 我仍在描述进度 → 浓缩更新「最新进度」。
  - **无论是否完成，此后都不再单独催这个任务。** 若下一个任务是学习/工作类型，就在**同一条消息**里回顾本任务进度、宣布切换、开始催下一个（避免发两条割裂的消息）。这条交接消息**即充当下一个任务的首次催促**——发出后把下一个任务直接置为「催促中」，监工便不会再对它单独重发首次催，因此不会重复。
- 到结尾仍未完成的任务**不滚动追问**，统一留到当天「日末」的总结里列出。

### 4.3 真实性判断的边界

催促开始与盯进度时，Bot 都靠 LLM 读我的回复来判断我是真在做事还是敷衍。Bot**没有任何独立监测手段**（如摄像头），只能依据对话内容，所以判断不保证准确。我可以随时用关键词或直接编辑文件来纠正状态。

### 4.4 多个任务同时活跃时

我尽量一次只推进一个任务，但偶尔会有重叠。设计本身允许并行——监工对每个任务独立判断，每条催促也都标明是哪个任务——但有几点必须注意：

- **回复归属（重要）**：当多个任务同时活跃、而我的回复没点明是哪个时（如只说「做完了」），若存在多个可能的匹配，Bot**反问澄清是哪个任务，绝不猜着改状态**。我想省事就直接用 `!已开始 / !已完成 <模糊代号>` 指明。
- **避免轰炸**：同一分钟若有多个任务都该发消息，合并成一条发出，不连发数条。
- **计划时间假设不重叠**：「计划时间」的尾取自下一个任务的时间点，这隐含任务按时间首尾相接。若我在 `plan.txt` 里写了时间重叠的任务，中点/结尾/交接的计算会失准——尽量避免，或在 `progress.csv` 里手动修正。
- **外出与学习/工作撞车（未来）**：外出要提前催「准备出门」，可能正好撞上某个进行中的学习/工作任务。该功能实现时需定义两者的优先级或合并方式；当前外出不触发，暂无此冲突。

### 4.5 消息语气

每条发给我的消息，语气由两个独立维度合成：

- **性格**：温柔鼓励 / 严厉施压 / 阴阳怪气 三选一，决定说话风味。默认**每个任务固定一个性格**（这个任务的所有消息都用它，不会忽冷忽热）；可在配置项一键切换成「每条消息重抽」或「每天固定一个」。
- **强度**：只在催促开始阶段随拖延时长递增（越拖越硬）；其余消息（进度确认、交接、日末总结、闲聊）用常态强度。

所以一条催促开始的消息 = 该任务的性格 + 随拖延递增的强度，例如「越来越辛辣的阴阳怪气」或「越来越急的温柔」。

---

## 5. 每日节奏

```
「日初」同步  →  白天各任务跑生命周期  →  「日末」总结 + 催计划  →（次日循环）
```

**「日末」**，Bot 做两件事：
1. **当日总结**：列出今天哪些任务完成、哪些没完成。
2. **催写计划**：直接催我写好/补全明天的计划（不检查 `plan.txt` 里现有内容，直接催）。我写好后，次日「日初」自动同步进 `progress.csv`。

## 6. 我的指令

| 输入 | 作用 |
|------|------|
| `!已开始 <模糊代号>` | 把任务标记为「进行中」，停止催开始 |
| `!已完成 <模糊代号>` | 把任务标记为「已完成」|
| `!重载` | 从 `plan.txt` 重新同步到 `progress.csv`（保留已有状态）|
| `!进度` | 把 `progress.csv` 渲染成表格发给我 |
| `!修改` | 让 Bot 修改 `progress.csv`（删任务、改时间等；只动这一个文件）|
| 其它任何消息 | 纯聊天，结合计划与进度回复，**不改任何文件** |

**模糊代号**：用 `!已开始` / `!已完成` 时不必逐字照抄代号，写个能认出是哪个任务的说法即可（如 `!已完成 PGM`、`!已完成 下午那个`），由 LLM 匹配到具体任务。多任务并行、指代不清时，Bot 会反问而不是猜（见 4.4）。

**`progress.csv` 何时会被改动**——仅三种情况：① 我手动编辑文件；② 我用了上表中会改文件的指令（`!已开始`、`!已完成`、`!重载`、`!修改`）；③ LLM 从对话中判断我确实开始/完成了某任务，自动改其状态（多任务歧义时先反问，见 4.4）。除此之外的对话一律是纯聊天，Bot 不擅自改文件。

## 7. 配置项（含默认值）

| 项 | 默认 |
|----|------|
| 名字（Bot 怎么自称）| `Bot`（我可以给它起名）|
| 日初（每天首次运行：从 `plan.txt` 同步到 `progress.csv` 的时刻）| 4:00 |
| 日末（每天收尾：当日总结 + 催写明天计划的时刻）| 23:00 |
| 催促退避曲线（间隔随拖延递增）| 头 5 分钟每 1 分钟；5–15 分钟每 3 分钟；15–30 分钟每 5 分钟；30 分钟以上每 10 分钟（最密受监工轮询间隔限制）|
| 监工轮询间隔 | 1 分钟 |
| 末位任务（没有下一个任务）的计划时间结尾 | 开始 +2 小时，封顶到「日末」|
| 性格粒度 | 默认「每个任务一个」；可切换「每条消息一个」或「每天一个」|

## 8. 不做什么

- 不支持多用户，只服务我一个人。
- 不能真正监测我是否在学习，只能靠对话 + LLM 判断。
- 「外出」类型的提前提醒：暂未实现，仅预留类型。
- 「其他」类型的任务不催，仅作时间分界。
- 不追踪历史趋势、不出周报（暂不做）。

## 9. 实现概要（技术选型，可替换）

**语言与布局**：Python 3.12+（3.13 可）。`src/` 布局，配置统一进 `pyproject.toml`，代码写类型注解。

**开发工具链**（2026 主流默认；若项目已在用别的就跟项目走，不擅自迁移）：

- 环境与依赖：**uv**（`uv add` 装依赖、`uv run` 跑命令、`uv sync` 同步环境；不手搓 venv、不直接 `pip install`）。
- Lint + 格式化：**ruff**（`ruff check --fix`、`ruff format`，一并替掉 black / flake8 / isort）。
- 测试：**pytest**。
- 类型检查：**mypy `--strict`**。（`ty` 很有前景但仍 beta、第三方 stub 不全，可本地试跑，CI 仍以 mypy 为准。`discord.py` 等库的 stub 不全时，局部 `# type: ignore` 即可，不必死磕。）
- 原则：能用「标准库 + 上面这套」解决就不加新依赖；其它库只在确实需要时才引入。

**运行时依赖（刻意压到最少）**：

- `discord.py` —— 收发私信，并用它自带的 `ext.tasks`（`@tasks.loop(minutes=1)`）来实现「监工」每分钟轮询，以及「日初」「日末」的定时触发；这样**省掉 APScheduler**。
- 一个 LLM 客户端（`openai` SDK 调 DeepSeek / Claude 的兼容接口，或直接 `httpx`）。
- CSV 用**标准库 `csv`**；配置用**环境变量 + 一个 dataclass**（不上 pydantic）。

**各部分落地**：

- 通信：Discord Bot，私信。
- 调度：`ext.tasks` 每分钟跑监工；日初 / 日末靠时间判断触发。
- 智能：调 LLM 承担——解析 `plan.txt`、判断任务类型、判断我回复的真实性、把进度浓缩成一句、生成语气文案（性格 × 强度，见 4.5）、把「模糊代号」匹配到具体任务（歧义则反问，见 4.4）。
- 存储：仅 `plan.txt` + `progress.csv`，无数据库。

**测试重点**：监工的决策（3.2 节那张「状态 + 时间 → 动作」表）是纯函数式、不碰 IO，最适合单元测试。本文一路记录的边界情形——中点 / 结尾不重复询问、催促到结尾即停（不无限催）、交接不重复催下一个、不依赖我及时回应——每条都应写成测试钉死，日后改动若破坏其一会立刻报警。CSV 读写一并测。Discord / LLM 这类 IO 难测，mock 或暂不覆盖。

**部署约束（重要坑）**：`progress.csv` 必须放在**重启后不会丢的存储**上。许多免费托管平台的文件系统是临时的（ephemeral），一重启就清空——若状态文件存在那里，Bot 半夜重启一次，所有任务的开始 / 进度 / 完成状态就全没了。务必用平台提供的**持久卷**（如 Railway Volume、Render Disk），或用文件不会丢的 VPS。

## 10. 待澄清 / 已知边界（评审记录）

> 一次设计评审的产物：把 SPEC 当状态机、压力测试它反复声明的不变式（**无记忆 / 重启不乱 / 手动编辑即时生效 / 不依赖及时回应 / 不重复轰炸**）后发现的缝隙。**这是过渡记录**：每条拍板后，把定论搬进上文相应小节，并从本节删除——一个事实只留一处。
>
> 分三类：**A = 明确缺陷/歧义（已给推荐改法）**；**B = 需产品负责人拍板的设计决定（保持开放）**；**C = 小问题，实现时定即可**。

### A 类：明确缺陷 / 歧义

- **A1 · 决策表行不互斥、无优先级（3.2）**。「进行中/已过中点」与「进行中/已过结尾」是两条独立行，正常运行按时间先后经过没问题；但若 Bot 在**整个中点窗口都宕机**，重启时已过结尾、且「最新进度」时间戳仍早于中点，**两行条件同时成立**而表未给优先级，可能去问一个早已过去的中点。**推荐**：明确优先级 **结尾 > 中点**（等价于给中点行加互斥条件「未过结尾」）。

- **A2 · 合并消息（4.4）与每任务固定性格（4.5）冲突**。同一分钟多个任务都该发时要「合并成一条」，但每个任务默认固定一个性格；当温柔任务与阴阳怪气任务需在同一条里合并，**这条用谁的性格未定义**。**推荐**：定一条规则（如「合并消息用全局/当天默认性格」或「按最拖延的那个任务的性格」），并在 4.4 / 4.5 交叉注明。*（含设计选择，倾向先记录。）*

- **A3 · 日初 / `!重载` 的合并语义只说「保留状态」，其余列未定义（3.1、6 节）**。一行 8 列，运行期产物不止「状态」：`实际时间`、`最新进度` 也是；`类型` 还可被手动改过（2 节）。重新解析 `plan.txt` 时这些列**保留还是被覆盖**没说清——只保留「状态」会清掉实际开始时间 / 最新进度，并覆盖手动修正的类型，破坏「手动编辑即时生效」。**推荐**：逐列定义合并策略——`状态 / 实际时间 / 最新进度` 一律保留；`任务 / 计划时间` 从 `plan.txt` 刷新；`类型` 见 **B1**。

- **A4 · 缺时区配置（§7）**。全程墙钟时间（`14:00`、日初 4:00、日末 23:00）+ 云端 24h 部署，但配置项无时区；云平台默认常为 UTC，会让所有催促整体偏移。**推荐**：§7 增加「时区」配置项（默认如 `Asia/Shanghai`），实现里所有时间比较都用它。

- **A5 · 时间戳塞在自由文本列（3.2、4.2）**。无记忆去重依赖解析「最新进度」里的时间戳，而该列是自由文本 + 用户可手动编辑；手动改进度若动坏时间戳格式，去重就失准（不再问 / 每分钟重问）。**推荐**：拆出独立的 `最新进度时间` 列（机器维护），`最新进度` 只放人读描述。

- **A6 · 外部编辑 vs 监工整文件 read-modify-write 的竞态（84 行）**。「手动改文件即时生效」是核心特性，但监工每分钟读整个 CSV→改→写回；若用户在「读」与「写」之间用外部编辑器保存，写回会覆盖该编辑（lost update）。discord.py 内部协程不真并发，但外部编辑器会。**推荐**：写回前检测文件 mtime/内容是否被外部改动（变了就重读再合并），或只重写发生变化的行，或缩小读-写窗口；SPEC 至少点出此约束。

- **A7 · 「进行中」初入时 `最新进度` 时间戳初值未定义（3.2、4.2）**。催促中→进行中时记 `实际时间` 头，但 `最新进度` 时间戳初值没定义，而中点去重正是「最新进度时间戳早于中点 → 发」。**推荐**：进入进行中时把 `最新进度时间` 置为**实际开始时间**，「早于中点」的语义自然成立。

- **A8 · 宕机/迟到时下一任务可能被双重催（4.2）**。交接消息「即充当下一任务的首次催促」并把下一任务直接置「催促中」，避免监工重复首催；但宕机重启后同一 tick 里既有「当前任务已过结尾待交接」又有「下一任务已过其开始时间、仍未开始」，**处理顺序未定义**——若先处理下一任务，监工按「未开始+到点」先发首催并置催促中，随后当前任务的结尾交接又「开始催下一个」→ 下一任务被催两次。**推荐**：定义同一 tick 内按计划时间先后处理；交接前检查下一任务是否已为「催促中」，已置则交接消息不再重复其首催。

### B 类：需产品负责人拍板（保持开放）

- **B1 · `!重载` 时手动改过的 `类型` 保留还是被 LLM 重判覆盖？**（承接 A3）手动把「学习」改成「外出」后重载，LLM 又判成「学习」——**听手动修正还是听 `plan.txt`？** 保留=尊重手动；覆盖=`plan.txt` 是唯一意图源。**待定。**

- **B2 · 「纯聊天」vs「LLM 自动改状态」的边界，默认是否「先确认再改」？**（6 节）单任务活跃时也可能误判（「我搞定了」可能是反讽）；多任务歧义已定为反问（4.4），但单任务误判无兜底。自动改前先发一句确认（更稳、更啰嗦）还是接受偶发误判（可手动纠正）？**待定。**

- **B3 · 跨天陈旧任务在 `progress.csv` 如何收尾？** 到结尾仍「进行中/催促中」的行保持原状留到日末；次日日初只解析当天任务，旧行被监工按日期过滤忽略（**非 bug**），但 CSV 会无限累积。是否在日初归档/清理昨天及更早的行（如移到 `history.csv`），还是放任累积？**待定。**

### C 类：小问题（实现时定即可）

- **退避曲线相位锚点（§7、4.1）**：「头 5 分钟每 1 分钟；5–15 每 3 分钟…」在边界分钟（t=5/15/30）和「每 3 分钟」的相位起点上不够精确。纯函数实现需一个确定的「哪些分钟偏移是催促分钟」定义，否则不同实现行为不一致。
- **代号碰撞**：`日期-时间-科目`（如 `0313-1400-PGM`）在同日同时段同科目时会撞；概率低，但非真唯一。
- **跨午夜任务**：`23:00-01:00` 这类跨日任务与日末 23:00 的关系未定义（可能本就超范围，确认即可）。
- **末位任务封顶**：开始 +2h 封顶到日末——若任务在 21:00 后开始，中点/结尾窗口被压缩，确认这是预期行为。
